Preliminary Honest Assessment
Before I write a single line of v3, let me be brutally honest about what v2 still gets wrong, what the judges at a national-level hackathon will actually test, and where the architecture has fundamental conceptual flaws — not just missing features.

Part 1: Deep Critique of v2 — What It Still Gets Wrong
Critique 1.1: The Query Router Is Architecturally Naive
The v2 problem:

QUERY_TYPE_PROMPT = """Classify this query into one of four types..."""
This is a hard routing system. It classifies a query into exactly one bucket and sends it down exactly one pipeline. This will catastrophically fail on the most interesting queries:

> "How has Apple's revenue changed over the last three quarters, and what does management say about the trend?"

This is simultaneously:

LOCAL_FACTUAL (three specific numbers)
MULTI_HOP (cross-quarter reasoning)
GLOBAL_THEMATIC (management tone/narrative)
Hard routing sends it to ONE pipeline. The answer will be incomplete every single time. A national-level judge will find this in the first 5 minutes.

What v2 also misses: The router uses the same LLM as the answer synthesizer. So for every query, you burn tokens on a classification call BEFORE doing any retrieval. If the classification is wrong (and it will be ~15% of the time on complex queries), the entire pipeline follows the wrong path with no recovery mechanism.

Critique 1.2: The Graph Schema Is Still Document-Centric, Not Knowledge-Centric
The v2 problem:

v2 adds EntityNodes, but the graph still fundamentally represents document structure rather than knowledge structure. The edges are:

ElementNode --[MENTIONS]--> EntityNode
ElementNode --[NEXT_IN_SEQUENCE]--> ElementNode
ElementNode --[ON_SAME_PAGE]--> ElementNode
This means:

You can find all elements that mention Apple
You cannot answer: "What is the relationship between Apple's revenue and its R&D spending?"
You cannot answer: "Which claim in section 3 contradicts the claim in section 7?"
You cannot answer: "What causal chain leads to the conclusion in the executive summary?"
The graph is a document index dressed up as a knowledge graph. Microsoft's GraphRAG paper (Edge et al. 2024) specifically distinguishes between element graphs (what v2 has) and knowledge graphs (what you need to win). The difference is whether edges carry semantic meaning about the relationship between entities, not just co-occurrence.

Critique 1.3: The ReAct Agent Has No Working Memory Architecture
The v2 problem:

self.conversation_history: list[ConversationTurn] = []
This is a turn buffer, not a working memory system. It has critical failure modes:

Context window overflow: After 10 turns, prepending 10 Q&A pairs will exceed context limits or crowd out evidence
No coreference resolution: "What about its competitor?" — the agent has no mechanism to resolve "its" and "competitor" against conversation history
No evidence reuse: Turn 3 retrieved 5 elements. Turn 4 asks a related question. v2 does a completely fresh retrieval — discarding perfectly relevant already-retrieved evidence
No contradiction detection: User says "assume revenue is $10B" in turn 2. Agent should flag if retrieved evidence contradicts this in turn 5. v2 never does this.
Critique 1.4: The Synthesis Engine Has a Fundamental Epistemics Problem
The v2 problem:

relevance = "HIGH" if score > 0.85 else "MEDIUM" if score > 0.65 else "LOW"
This weights evidence by retrieval score (embedding cosine similarity). But retrieval score ≠ epistemic confidence. Consider:

A sentence that directly contradicts the correct answer can score 0.95 if it's semantically similar to the query
The actual correct answer might be buried in a low-scoring element that was retrieved via graph traversal, not vector search
The synthesizer should weight evidence by grounding strength (how many other elements corroborate it), not raw retrieval score. v2 never computes corroboration.

Critique 1.5: Community Detection Is Computed at Ingestion Time — This Will Break
The v2 problem:

communities = nx_comm.louvain_communities(undirected, seed=42)
Louvain is run once at ingestion. Community summaries are static. This means:

Multi-document scenario: User ingests Document A (Apple 10-K). Louvain runs. Communities formed. User then ingests Document B (Apple earnings call). Now the graph changes fundamentally — but communities are never recomputed. The cross-document thematic structure is permanently wrong.
Incremental ingestion is broken by design. This will fail during the demo if the judge asks you to add a second document.
Critique 1.6: The Evaluation Harness Measures the Wrong Things
The v2 problem:

correct = sum(1 for r in results if r["expected"].lower()[:40] in r["got"].lower())
This is substring matching. It will:

Mark "$10.2 billion" as wrong when the expected answer is "$10,200,000,000"
Mark "Q3 revenue grew 12%" as wrong when expected is "revenue increased by 12% in the third quarter"
Give zero credit for partially correct answers
For a national hackathon, this accuracy metric will either overcount (false positives) or undercount (false negatives) — making your benchmark numbers unreliable under scrutiny. Judges will ask "how do you measure accuracy?" and this will not hold up.

Critique 1.7: Three Missing Capabilities That Judges Will Specifically Test
Missing Capability A — Numerical Reasoning: Financial documents require arithmetic. "What was the year-over-year revenue growth?" is not answerable by retrieval alone — it requires computing (Q3_2024 - Q3_2023) / Q3_2023 * 100. No RAG system without a computation layer can answer this correctly.

Missing Capability B — Negative Space Queries: "Does this document mention any risks related to supply chain?" requires the system to reason about what is not present, or what is present but framed negatively. Standard vector search retrieves documents similar to "supply chain risks" — but if the document says "we have no supply chain concerns," a naive system will retrieve it and misinterpret it as confirming supply chain risks exist.

Missing Capability C — Temporal Reasoning: "What changed between the Q2 and Q3 reports?" requires the system to maintain document-level temporal metadata AND perform differential comparison. v2 has no temporal awareness whatsoever.

Part 2: The Fundamental Architectural Insight v2 Misses
Here is the core insight that separates a good RAG system from a championship RAG system:

> The retrieval problem and the reasoning problem are not sequential — they are interleaved. The system must reason about what it doesn't know and retrieve specifically to fill those gaps, in a tight loop, not a linear pipeline.

v2's architecture is:

Query → Route → Retrieve → Reason → Answer
This is still fundamentally linear. The championship architecture is:

Query → Decompose → [Retrieve → Reason → Identify Gaps → Re-Retrieve → Reason]* → Verify → Answer
The difference is the feedback loop between reasoning and retrieval. The agent should actively reason about the structure of its own ignorance and direct retrieval accordingly.

This is what RETRO (Borgeaud et al.), Self-RAG (Asai et al. 2023), and FLARE (Jiang et al. 2023) all point toward — but v2 implements none of them correctly.

Part 3: v3 Championship Architecture
Core Design Principles
Principle 1: SOFT ROUTING over HARD ROUTING
           — Every query activates multiple retrieval strategies in parallel
           — Results are fused, not selected

Principle 2: KNOWLEDGE GRAPH over DOCUMENT GRAPH  
           — Edges carry semantic relationship types
           — Supports causal, temporal, contradictory relationships

Principle 3: ACTIVE RETRIEVAL over PASSIVE RETRIEVAL
           — Agent identifies what it doesn't know
           — Retrieval is targeted at specific epistemic gaps

Principle 4: EPISTEMIC CONFIDENCE over RETRIEVAL SCORE
           — Evidence weighted by corroboration count
           — Contradictions are surfaced, not hidden

Principle 5: COMPUTATION AS A FIRST-CLASS TOOL
           — Numerical reasoning via code execution
           — Not all questions can be answered by text retrieval

Principle 6: INCREMENTAL CONSISTENCY
           — All indices, graphs, and communities support incremental updates
           — Multi-document ingestion is a first-class operation

@dataclass
class RetrievalStrategy:
    """A single retrieval strategy with weight and parameters."""
    strategy_type: str          # "vector", "graph_entity", "community", 
                                # "temporal", "numerical", "cross_document"
    weight: float               # 0.0 - 1.0, sum across strategies ≈ 1.0
    parameters: dict            # strategy-specific parameters
    sub_queries: list[str]      # decomposed sub-questions for this strategy

@dataclass  
class QueryPlan:
    """Complete execution plan for a query."""
    original_query: str
    strategies: list[RetrievalStrategy]
    requires_computation: bool
    requires_temporal_comparison: bool
    requires_cross_document: bool
    ambiguous_references: list[str]   # ["its", "the company"] → need resolution

Rules:
- Weight sum should equal 1.0
- Complex queries MUST have multiple strategies

class IncrementalGraphBuilder:
    """
    Builds and maintains the knowledge graph with full support for
    incremental document addition. Communities are recomputed only
    for affected subgraphs, not the entire graph.
    """

    def __init__(self, falkordb_client, llm_client, knowledge_extractor):
        self.db = falkordb_client
        self.llm = llm_client
        self.extractor = knowledge_extractor
        self.G = nx.DiGraph()  # In-memory mirror for community analysis

    def ingest_document(
        self, 
        doc_id: str, 
        elements: list[DocumentElement]
    ) -> Generator[dict, None, None]:
        """
        Streaming ingestion with incremental graph updates.
        Yields progress events throughout.
        """
        yield {"phase": "parsing", "progress": 0.0, "message": "Starting ingestion"}

        # Phase 1: Element and chunk nodes
        all_chunks = []
        for i, el in enumerate(elements):
            chunks = self._semantic_chunk(el)
            all_chunks.extend(chunks)
            self._add_element_node(el)
            self._add_structural_edges(el, elements)
            yield {
                "phase": "structuring", 
                "progress": 0.1 + (i / len(elements)) * 0.2,
                "message": f"Structured element {i+1}/{len(elements)}: {el.element_type}"
            }

        # Phase 2: Knowledge extraction (claims, metrics, events)
        yield {"phase": "extracting", "progress": 0.3, "message": "Extracting knowledge"}

        all_claims = []
        all_metrics = []
        for i, chunk in enumerate(all_chunks):
            extracted = self.extractor.extract(chunk)
            claim_nodes = self._add_claim_nodes(extracted["claims"], chunk)
            metric_nodes = self._add_metric_nodes(extracted["metrics"], chunk)
            all_claims.extend(claim_nodes)
            all_metrics.extend(metric_nodes)
            yield {
                "phase": "extracting",
                "progress": 0.3 + (i / len(all_chunks)) * 0.2,
                "message": f"Extracted from chunk {i+1}/{len(all_chunks)}"
            }

        # Phase 3: Entity extraction and cross-document linking
        yield {"phase": "entity_linking", "progress": 0.5, "message": "Linking entities"}

        entities = self._extract_and_link_entities(all_chunks)
        cross_doc_edges = self._find_cross_document_connections(entities)
        yield {
            "phase": "entity_linking",
            "progress": 0.6,
            "message": f"Found {len(cross_doc_edges)} cross-document connections"
        }

        # Phase 4: Metric comparison edges (temporal reasoning backbone)
        yield {"phase": "temporal", "progress": 0.65, "message": "Building temporal graph"}

        self._build_metric_comparison_edges(all_metrics)
        self._build_temporal_event_chain()

        # Phase 5: Contradiction detection across document
        yield {"phase": "validation", "progress": 0.70, "message": "Detecting contradictions"}

        contradiction_count = self._detect_cross_claim_contradictions(all_claims)
        yield {
            "phase": "validation",
            "progress": 0.75,
            "message": f"Found {contradiction_count} internal contradictions"
        }

        # Phase 6: INCREMENTAL community detection
        # Key insight: Only recompute communities for the subgraph 
        # reachable from newly added nodes within 2 hops
        yield {"phase": "communities", "progress": 0.80, "message": "Detecting communities"}

        affected_nodes = self._get_affected_subgraph(doc_id)
        updated_communities = self._incremental_louvain(affected_nodes)
        community_summaries = self._generate_community_summaries(updated_communities)

        yield {
            "phase": "communities",
            "progress": 0.90,
            "message": f"Identified {len(updated_communities)} communities"
        }

        # Phase 7: Document-level summary with cross-document awareness
        doc_summary = self._generate_document_summary(doc_id, elements, community_summaries)

        yield {
            "phase": "complete",
            "progress": 1.0,
            "message": f"Ingestion complete",
            "stats": {
                "elements": len(elements),
                "chunks": len(all_chunks),
                "claims": len(all_claims),
                "metrics": len(all_metrics),
                "communities": len(updated_communities),
                "cross_doc_edges": len(cross_doc_edges),
                "contradictions": contradiction_count
            }
        }

    def _incremental_louvain(self, affected_nodes: set) -> dict:
        """
        CRITICAL: Only recompute communities for affected subgraph.
        Preserve community assignments for unaffected nodes.
        """
        # Extract 3-hop neighborhood of new nodes
        affected_subgraph = nx.ego_graph(
            self.G.to_undirected(), 
            nbunch=affected_nodes,
            radius=3
        )

        # Run Louvain only on affected subgraph
        if len(affected_subgraph.nodes) > 0:
            new_communities = nx_comm.louvain_communities(
                affected_subgraph, seed=42
            )

            # Merge: preserve unaffected, update affected
            for comm_id, community in enumerate(new_communities):
                for node_id in community:
                    # Use globally unique community IDs to avoid conflicts
                    global_comm_id = f"{self._get_next_community_id()}_{comm_id}"
                    nx.set_node_attributes(
                        self.G, 
                        {node_id: global_comm_id}, 
                        "community"
                    )

        return self._get_all_communities()

    def _build_metric_comparison_edges(self, metrics: list[MetricNode]) -> None:
        """
        Build COMPARED_TO edges between metrics of the same entity
        across different time periods. This is the backbone of temporal reasoning.
        """
        from itertools import combinations

        # Group metrics by (entity, metric_name)
        metric_groups = {}
        for metric in metrics:
            key = (metric.entity_id, metric.metric_name)
            metric_groups.setdefault(key, []).append(metric)

        # Create comparison edges for all pairs within same group
        for (entity_id, metric_name), group in metric_groups.items():
            for m1, m2 in combinations(group, 2):
                # Add COMPARED_TO edge with computed delta
                delta = None
                if m1.unit == m2.unit:
                    delta = m2.value - m1.value
                    pct_change = delta / m1.value * 100 if m1.value != 0 else None

                self._add_edge(
                    m1.metric_id, m2.metric_id, 
                    EdgeType.COMPARED_TO,
                    properties={
                        "delta": delta,
                        "pct_change": pct_change,
                        "from_period": m1.period,
                        "to_period": m2.period
                    }
                )

    def _detect_cross_claim_contradictions(
        self, claims: list[ClaimNode]
    ) -> int:
        """
        Detect claims that directly contradict each other.
        Uses semantic similarity + logical negation patterns.
        """
        contradiction_count = 0

        # Group claims by involved entities for efficient comparison
        entity_claims = {}
        for claim in claims:
            for entity_id in claim.entities_involved:
                entity_claims.setdefault(entity_id, []).append(claim)

        # Check pairs of claims about the same entity
        for entity_id, claim_group in entity_claims.items():
            if len(claim_group) < 2:
                continue

            for i, claim1 in enumerate(claim_group):
                for claim2 in claim_group[i+1:]:
                    if self._claims_contradict(claim1, claim2):
                        self._add_edge(
                            claim1.claim_id,
                            claim2.claim_id,
                            EdgeType.CONTRADICTS,
                            properties={"confidence": 0.85}
                        )
                        contradiction_count += 1

        return contradiction_count

    def _claims_contradict(
        self, claim1: ClaimNode, claim2: ClaimNode
    ) -> bool:
        """
        Detect logical contradiction between two claims.
        Uses numerical conflict detection for metrics.
        """
        # Numerical conflict: same metric, different values, same period
        if (claim1.temporal_scope == claim2.temporal_scope and
            claim1.temporal_scope is not None):
            for v1 in claim1.numerical_values:
                for v2 in claim2.numerical_values:
                    if (v1["unit"] == v2["unit"] and
                        abs(v1["value"] - v2["value"]) / 
                        max(abs(v1["value"]), 1) > 0.05):  # 5% tolerance
                        return True

        return False

import asyncio
from concurrent.futures import ThreadPoolExecutor

class ParallelHybridRetrievalEngine:
    """
    Executes multiple retrieval strategies in PARALLEL (not sequential)
    and fuses results using strategy-weight-aware RRF.
    """

    def __init__(self, qdrant_client, falkordb_client, reranker, llm_client):
        self.qdrant = qdrant_client
        self.graph = falkordb_client
        self.reranker = reranker
        self.llm = llm_client
        self.executor = ThreadPoolExecutor(max_workers=6)

    async def retrieve(
        self, plan: QueryPlan, top_k_final: int = 8
    ) -> "RetrievalResult":
        """
        Execute all strategies in parallel, fuse results.
        """
        # Build async tasks for each strategy
        tasks = []
        for strategy in plan.strategies:
            task = self._execute_strategy_async(strategy, plan)
            tasks.append((strategy, task))

        # Execute all in parallel
        strategy_results = {}
        for strategy, task in tasks:
            results = await task
            strategy_results[strategy.strategy_type] = {
                "results": results,
                "weight": strategy.weight
            }

        # Fuse all results using weighted RRF
        fused = self._weighted_rrf_fusion(strategy_results)

        # Retrieve full element objects for top candidates
        top_candidates = self._fetch_elements(fused[:25])

        # Cross-encoder re-rank
        reranked = self._cross_encoder_rerank(
            plan.original_query, top_candidates, top_n=top_k_final
        )

        # Corroboration scoring (the key epistemic upgrade from v2)
        corroborated = self._compute_corroboration_scores(reranked, strategy_results)

        return RetrievalResult(
            elements=corroborated,
            strategies_used=list(strategy_results.keys()),
            fusion_scores=fused[:top_k_final],
            plan=plan
        )

    async def _execute_strategy_async(
        self, strategy: RetrievalStrategy, plan: QueryPlan
    ) -> list[dict]:
        """Dispatch to appropriate retrieval method."""

        loop = asyncio.get_event_loop()

        if strategy.strategy_type == "vector":
            return await loop.run_in_executor(
                self.executor,
                self._vector_search,
                strategy.sub_queries, 20
            )

        elif strategy.strategy_type == "graph_entity":
            return await loop.run_in_executor(
                self.executor,
                self._graph_entity_search,
                strategy.parameters.get("entities", []), 
                strategy.parameters.get("depth", 2)
            )

        elif strategy.strategy_type == "community":
            return await loop.run_in_executor(
                self.executor,
                self._community_search,
                plan.original_query
            )

        elif strategy.strategy_type == "temporal":
            return await loop.run_in_executor(
                self.executor,
                self._temporal_search,
                strategy.parameters.get("time_range"),
                strategy.parameters.get("entities", [])
            )

        elif strategy.strategy_type == "numerical":
            return await loop.run_in_executor(
                self.executor,
                self._metric_graph_search,
                strategy.parameters.get("metrics", []),
                strategy.parameters.get("entities", [])
            )

        elif strategy.strategy_type == "cross_document":
            return await loop.run_in_executor(
                self.executor,
                self._cross_document_search,
                strategy.sub_queries
            )

        return []

    def _metric_graph_search(
        self, metric_names: list[str], entities: list[str]
    ) -> list[dict]:
        """
        Query the MetricNode subgraph specifically.
        Critical for numerical reasoning queries.
        """
        results = []

        for entity in entities:
            for metric_name in metric_names:
                # Find all MetricNodes for this entity/metric combination
                query = """
                MATCH (m:METRIC {entity_id: $entity_id, metric_name: $metric_name})
                OPTIONAL MATCH (m)-[r:COMPARED_TO]->(m2:METRIC)
                RETURN m, r, m2
                ORDER BY m.period DESC
                LIMIT 10
                """
                rows = self.graph.query(
                    query, 
                    {"entity_id": entity, "metric_name": metric_name}
                )

                for row in rows:
                    results.append({
                        "element_id": row["m"]["source_element_id"],
                        "metric_data": {
                            "value": row["m"]["value"],
                            "unit": row["m"]["unit"],
                            "period": row["m"]["period"],
                            "compared_to": row.get("m2"),
                            "delta": row.get("r", {}).get("delta"),
                            "pct_change": row.get("r", {}).get("pct_change")
                        },
                        "retrieval_type": "metric_graph"
                    })

        return results

    def _temporal_search(
        self, time_range: Optional[str], entities: list[str]
    ) -> list[dict]:
        """
        Query EventNode chain for temporal reasoning.
        """
        query = """
        MATCH (e1:EVENT)-[r:TEMPORALLY_BEFORE]->(e2:EVENT)
        WHERE any(ent IN e1.entities_involved WHERE ent IN $entities)
        RETURN e1, e2, r
        ORDER BY e1.event_date
        LIMIT 20
        """
        rows = self.graph.query(query, {"entities": entities})

        results = []
        for row in rows:
            results.append({
                "element_id": row["e1"]["source_element_id"],
                "event_data": {
                    "before": row["e1"]["description"],
                    "after": row["e2"]["description"],
                    "date": row["e1"]["event_date"]
                },
                "retrieval_type": "temporal_chain"
            })

        return results

    def _weighted_rrf_fusion(
        self, 
        strategy_results: dict,
        k: int = 60
    ) -> list[dict]:
        """
        RRF fusion where each strategy's contribution is weighted
        by its planned weight from the QueryPlan.
        """
        scores: dict[str, float] = {}
        metadata: dict[str, dict] = {}

        for strategy_type, data in strategy_results.items():
            weight = data["weight"]
            for rank, result in enumerate(data["results"]):
                elem_id = result["element_id"]

                # Weighted RRF: multiply RRF score by strategy weight
                rrf_score = weight * (1 / (k + rank + 1))
                scores[elem_id] = scores.get(elem_id, 0) + rrf_score

                # Preserve richest metadata
                if elem_id not in metadata:
                    metadata[elem_id] = {}
                metadata[elem_id].update(result.get("metric_data", {}))
                metadata[elem_id].update(result.get("event_data", {}))
                metadata[elem_id]["strategies_hit"] = (
                    metadata[elem_id].get("strategies_hit", []) + [strategy_type]
                )

        ranked = sorted(scores.keys(), key=lambda x: scores[x], reverse=True)
        return [{"element_id": eid, "fusion_score": scores[eid], **metadata[eid]} 
                for eid in ranked]

    def _compute_corroboration_scores(
        self, 
        elements: list[DocumentElement],
        strategy_results: dict
    ) -> list[tuple[DocumentElement, float, dict]]:
        """
        Weight evidence by how many independent retrieval strategies
        surfaced it — this is epistemic confidence, not retrieval score.
        """
        strategy_sets = {}
        for strategy_type, data in strategy_results.items():
            strategy_sets[strategy_type] = {r["element_id"] for r in data["results"]}

        scored = []
        for element in elements:
            # Count how many independent strategies returned this element
            strategies_corroborating = sum(
                1 for s_set in strategy_sets.values() 
                if element.element_id in s_set
            )

            # Corroboration score: 0.5 base + 0.1 per additional corroborating strategy
            corroboration_score = 0.5 + (strategies_corroborating - 1) * 0.1
            corroboration_score = min(corroboration_score, 1.0)

            scored.append((
                element,
                corroboration_score,
                {"strategies_count": strategies_corroborating}
            ))

        return sorted(scored, key=lambda x: x[1], reverse=True)

import ast
import operator
import math
from typing import Any

SAFE_OPERATORS = {
    ast.Add: operator.add,
    ast.Sub: operator.sub,
    ast.Mult: operator.mul,
    ast.Div: operator.truediv,
    ast.Pow: operator.pow,
    ast.Mod: operator.mod,
}

class SafeExpressionEvaluator:
    """
    Evaluates mathematical expressions safely (no eval(), no exec()).
    Used by the ComputationalReasoningTool.
    """

    def evaluate(self, expression: str) -> float:
        tree = ast.parse(expression, mode='eval')
        return self._eval_node(tree.body)

    def _eval_node(self, node):
        if isinstance(node, ast.Constant):
            return node.value
        elif isinstance(node, ast.BinOp):
            left = self._eval_node(node.left)
            right = self._eval_node(node.right)
            return SAFE_OPERATORS[type(node.op)](left, right)
        elif isinstance(node, ast.UnaryOp) and isinstance(node.op, ast.USub):
            return -self._eval_node(node.operand)
        elif isinstance(node, ast.Call):
            func_name = node.func.id if isinstance(node.func, ast.Name) else None
            if func_name in {"abs", "round", "sqrt", "log"}:
                args = [self._eval_node(a) for a in node.args]
                return getattr(math, func_name)(*args) if func_name in {"sqrt", "log"} \
                    else __builtins__[func_name](*args)
        raise ValueError(f"Unsafe operation: {ast.dump(node)}")


class ComputationalReasoningTool:
    """
    Solves numerical queries by:
    1. Extracting relevant MetricNodes from the graph
    2. Generating a computation plan
    3. Executing the computation safely
    4. Returning result with full audit trail
    """

    def __init__(self, graph_client, llm_client):
        self.graph = graph_client
        self.llm = llm_client
        self.evaluator = SafeExpressionEvaluator()

    def solve(self, query: str, available_metrics: list[MetricNode]) -> dict:
        """
        Given a numerical query and available metric nodes,
        generate and execute a computation plan.
        """

        if not available_metrics:
            return {"success": False, "reason": "No metrics available"}

        # Generate computation plan
        plan_prompt = f"""Given these available numerical metrics:
{self._format_metrics(available_metrics)}

Write a computation plan to answer: {query}

Return JSON:
{{
  "variables": {{"var_name": metric_id, ...}},
  "expression": "mathematical expression using var names",
  "unit": "result unit",
  "explanation": "what this computes and why"
}}"""

        response = self.llm.chat.completions.create(
            model="llama-3.1-8b-instant",
            messages=[{"role": "user", "content": plan_prompt}],
            temperature=0.0,
            response_format={"type": "json_object"}
        )

        plan = json.loads(response.choices[0].message.content)

        # Bind variables to actual values
        variable_values = {}
        variable_provenance = {}

        for var_name, metric_id in plan["variables"].items():
            metric = next(
                (m for m in available_metrics if m.metric_id == metric_id), None
            )
            if metric is None:
                return {
                    "success": False, 
                    "reason": f"Metric {metric_id} not found"
                }
            variable_values[var_name] = metric.value
            variable_provenance[var_name] = {
                "metric_name": metric.metric_name,
                "entity": metric.entity_id,
                "period": metric.period,
                "source": metric.source_element_id,
                "unit": metric.unit
            }

        # Substitute variables into expression
        expression = plan["expression"]
        for var_name, value in variable_values.items():
            expression = expression.replace(var_name, str(value))

        # Safe evaluation
        try:
            result = self.evaluator.evaluate(expression)
            return {
                "success": True,
                "result": result,
                "unit": plan["unit"],
                "expression": plan["expression"],
                "variable_values": variable_provenance,
                "explanation": plan["explanation"],
                "computed_expression": expression
            }
        except Exception as e:
            return {"success": False, "reason": f"Computation error: {str(e)}"}

    def _format_metrics(self, metrics: list[MetricNode]) -> str:
        lines = []
        for m in metrics:
            lines.append(
                f"- ID: {m.metric_id} | {m.entity_id}.{m.metric_name} "
                f"= {m.value} {m.unit} | Period: {m.period}"
            )
        return "\n".join(lines)
